# Atomsyn V2.x — Chat Module Full Redesign

## Context

聊天模块是 Atomsyn 面向**非技术用户**的核心入口——CLI 用户通过 atomsyn-read/write/mentor 三个 skill 在 Claude Code/Cursor 中操作知识库，但普通用户需要一个 GUI 聊天界面来获得同等能力。当前 ChatPage 只有基础的单轮 Copilot + 沉淀分类，需要全面重构为：流式 Markdown 输出 + skill 驱动对话 + 会话管理 + 记忆系统 + 上下文 harness + 文件/图片上传 + / 命令 + 模型选择器。

### 关键决策 (已确认)
- **沉淀策略**: 允许 skill 驱动沉淀 (atomsyn-write)，但需用户确认
- **流式方案**: Markdown + 卡片协议 `[[atom:id]]`，聊天专用 LLM streaming client，不影响其他模块
- **记忆粒度**: 用户偏好 + 关键决策 (long-term memory)
- **交付范围**: 全量一次交付

## 3 · SOUL.md 设计

```markdown
# SOUL.md — Atomsyn Copilot

> 这个文件定义了 Atomsyn 内置 AI 助手的灵魂。
> 用户可以编辑此文件来个性化 AI 的行为方式。

## 核心身份

- **名字**: Atomsyn Copilot
- **定位**: 认知双向操作系统的内置 AI 助手
- **使命**: 帮助用户沉淀、检索、连接和反思自己的认知资产

## 产品语境

Atomsyn 是一个认知的双向操作系统:
- **仓库层**: 沉淀 + 唤醒——方法论、经验碎片、笔记
- **结构层**: 让碎片长出骨架——方法论框架 × 经验碎片相遇
- **教练层**: 主动推动复盘、指出盲区、建议行动

北极星: **让你积累的认知，在需要时醒来。**

## 沟通风格

- 用「你」而非「您」，平等对话
- 简洁精准，不说废话
- 引用数据时标注来源 (覆盖率、碎片数、天数)
- 推荐方法论/经验时，给出具体理由而非泛泛而谈
- 当知识库无匹配时，诚实说「暂无匹配」
- 教练模式下：镜子而非老师，轻轻推而非居高临下

## 价值观

- **100% 本地主权**: 用户数据从不离开本地
- **使用即维护**: 碎片被检索、被引用的那一刻就是被维护的那一刻
- **认知复利**: 今天的沉淀是明天的武器
```

## 4 · AGENTS.md 设计

```markdown
# AGENTS.md — Atomsyn Chat Agent 行为规范

> 这个文件定义了聊天 Agent 的工具使用规则和 skill 路由逻辑。

## 可用 Skills

你可以在对话中调用以下 skill 能力:

### atomsyn-read (知识检索)
- **触发**: 用户问到方法论、经验、知识盲区，或任何可能与知识库相关的问题
- **行为**: 在回复前先搜索知识库，将匹配结果以卡片形式嵌入回复
- **卡片语法**: `[[atom:atom_id|显示名称]]` — 前端会渲染为可点击的知识卡片
- **原则**: 主动但安静——搜索后如无匹配，不需要告诉用户"我搜了但没找到"

### atomsyn-write (经验沉淀)
- **触发**: 用户明确说"记下来/沉淀/remember/crystallize"，或对话中产生了值得保存的洞察
- **行为**: 提取核心信息，构造经验碎片，请用户确认后沉淀
- **必须字段**: name, insight, sourceContext, role, situation, activity, insight_type, tags
- **原则**: 永远先征求用户确认，不自动写入

### atomsyn-mentor (认知复盘)
- **触发**: 用户说"复盘/导师模式/看看我的盲区/我最近学了什么"
- **行为**: 调取分析数据，生成复盘报告 (优势区域 + 盲区警示 + 行动建议)
- **后续**: 可深入某个盲区、制定学习计划、引导沉淀新经验

## 回复规范

### 格式
- 使用自然 Markdown 回复 (支持标题、列表、粗体、代码块)
- 知识卡片用 `[[atom:atom_id|名称]]` 嵌入
- 沉淀确认用 `[[ingest:confirm]]` 触发确认 UI
- 不要输出原始 JSON 给用户

### 上下文感知
- 你会收到用户的历史对话摘要 (最近 2-3 轮)
- 你会收到用户的长期记忆 (偏好、关键决策)
- 你会收到知识库索引 (方法论 + 经验碎片摘要)
- 根据用户问题判断是否需要调用 skill

### 安全规则
- 不编造知识库中不存在的方法论
- 不自动执行写入操作
- 不泄露用户隐私数据到外部
```

## 5 · 流式 LLM Client (chat 专用)

### 5.1 新文件: `src/lib/chatLlmClient.ts`

不修改现有 `llmClient.ts`，新建 chat 专用的流式 client。

```typescript
// 核心接口
interface ChatStreamOptions {
  messages: ChatMessage[]        // 完整对话历史 (已裁剪)
  systemPrompt: string           // SOUL + AGENTS + context
  modelConfig: ModelConfig       // 当前选中模型
  apiKey: string
  onToken: (token: string) => void    // 流式 token 回调
  onComplete: (fullText: string) => void
  onError: (error: Error) => void
  signal?: AbortSignal           // 支持取消
}

interface ChatMessage {
  role: 'user' | 'assistant' | 'system'
  content: string | ContentPart[]  // 支持图片
}

type ContentPart = 
  | { type: 'text'; text: string }
  | { type: 'image'; data: string; mediaType: string }
```

### 5.2 双分支 streaming

- **Anthropic**: `client.messages.stream()` → onText 回调
- **OpenAI-compatible**: `fetch(url, { stream: true })` → SSE 解析 `data: {...}` → delta.content

### 5.3 卡片协议

AI 输出中的 `[[atom:atom_xxx|方法论名称]]` 被前端 Markdown 渲染器拦截，替换为可点击的知识卡片组件。

AI 输出中的 `[[ingest:confirm|{json}]]` 被前端拦截，渲染为沉淀确认卡片 (预览 + 确认/取消按钮)。

## 6 · 上下文 Harness 设计

### 6.1 上下文分层

每次 LLM 调用的 system prompt 由以下层组成 (按优先级):

```
Layer 1: SOUL.md (固定, ~300 tokens)
Layer 2: AGENTS.md (固定, ~500 tokens)  
Layer 3: Skill 索引 (按需, ~200 tokens)
  - 可用 skill 名称 + 触发条件摘要
Layer 4: 知识库快照 (按需, ~1000 tokens)
  - 方法论索引 top-N (名称+ID+tags)
  - 近期经验碎片 top-10 (名称+摘要)
Layer 5: 用户记忆 (按需, ~200 tokens)
  - 从 memories.jsonl 加载相关记忆
Layer 6: 对话历史 (动态裁剪)
  - 最近 2-3 轮完整消息
  - 更早的消息压缩为摘要
```

### 6.2 上下文裁剪策略

```
总预算: ~4000 tokens (system) + ~2000 tokens (history)

1. SOUL + AGENTS: 固定注入 (~800 tokens)
2. Skill 索引: 固定注入 (~200 tokens)
3. 知识库快照: 按相关性注入
   - 如果用户消息包含方法论关键词 → 注入匹配的 top-5 atoms
   - 否则注入最近更新的 top-10
4. 记忆: 全量注入 (用户偏好通常不超过 20 条)
5. 对话历史:
   - 最近 3 轮: 完整保留
   - 第 4-6 轮: 截断到每条 200 字
   - 第 7+ 轮: 丢弃 (依赖会话摘要)
6. 会话超过 6 轮时, 在 system 层注入前 6 轮的 1 句话摘要
```

### 6.3 摘要生成

当对话超过 6 轮时，在用户发送新消息前，异步调用 LLM 生成前 N 轮的摘要 (1-2 句话)，缓存到 session 中。

## 7 · 会话管理

### 7.1 Session 数据结构

```typescript
interface ChatSession {
  id: string                    // sess_<timestamp>
  title: string                 // 自动从首条消息生成, 用户可重命名
  createdAt: string
  updatedAt: string
  modelId?: string              // 创建时使用的模型
  messages: ChatMessageRecord[]
  summary?: string              // 6轮后的摘要缓存
}

interface ChatMessageRecord {
  id: string
  role: 'user' | 'assistant'
  content: string               // 原始文本 (含卡片标记)
  timestamp: string
  attachments?: Attachment[]    // 图片/文件
  metadata?: {
    model?: string
    tokens?: number
    skillsUsed?: string[]
  }
}

interface SessionIndex {
  sessions: SessionIndexEntry[]
}

interface SessionIndexEntry {
  id: string
  title: string
  updatedAt: string
  messageCount: number
  preview: string               // 最后一条消息的前 80 字
}
```

### 7.2 存储 API

在 `vite-plugin-data-api.ts` 新增端点:
```

GET    /api/chat/soul              → 读取 SOUL.md
```

## 8 · 记忆系统

### 8.1 记忆数据结构

```typescript
interface MemoryEntry {
  id: string                    // mem_<timestamp>
  type: 'preference' | 'decision' | 'context'
  content: string               // 简洁的一句话
  source: 'auto' | 'user'      // auto=AI自动提取, user=用户主动保存
  sessionId: string
  createdAt: string
}
```

### 8.2 自动提取策略

在每轮 AI 回复完成后，异步调用一个轻量 LLM prompt:

```
给定以下对话，提取值得长期记住的用户偏好或关键决策 (如有)。
如果没有值得记忆的内容，返回空数组。
只记录用户明确表达的偏好、决策、反馈。不记录临时状态。
格式: [{"type": "preference|decision", "content": "一句话"}]
```

- 使用 haiku 级别模型 (或当前模型的低 token 调用)
- 后台异步执行，不阻塞对话
- 每轮最多提取 2 条记忆

### 8.3 记忆加载

每次构建 context 时:
1. 读取全部 memories.jsonl
2. 去重 (content 相似度 > 0.8 的取最新)
3. 按 type 分组注入 system prompt

## 9 · Markdown 渲染 + 卡片系统

### 9.1 渲染管线

```
AI 流式 token → 累积文本 → Markdown 解析 → 自定义渲染器
                                               │
                                    ┌──────────┴──────────┐
                                    │ 正常 MD 元素         │ → 标准渲染
                                    │ [[atom:id|name]]    │ → AtomChip 组件
                                    │ [[ingest:confirm|…]]│ → IngestConfirmCard
                                    └─────────────────────┘
```

### 9.2 技术选型

- **Markdown 渲染: `streamdown`** (Vercel 出品，专为 LLM 流式输出设计)
  - 原生流式支持——优雅处理未闭合的 markdown 标记 (粗体/代码块/列表流式中不会闪烁)
  - 内置 Shiki 代码高亮 (VS Code 同款引擎，100+ 语言)
  - 内置 KaTeX 数学公式 + Mermaid 图表
  - 支持自定义 remark 插件——`[[atom:id]]` 卡片协议可无缝接入
  - ~1M 周下载量，Vercel 官方维护
- **卡片协议**: 自定义 remark 插件，匹配 `[[...]]` 语法，注册为 streamdown 的 custom component
- **Chat Shell**: 自建 (Framer Motion + Tailwind)，保持 Linear/Raycast 视觉一致性
- **不用** assistant-ui (架构强约束，和 Zustand + 自定义设计语言冲突)
- **不用** @ant-design/x (视觉体系差异大，包体积重)
- **不用** react-markdown (不支持流式，streaming 时闪烁)

### 9.3 AtomChip 组件

点击后跳转到对应 atom 详情页。显示:
- atom 名称
- kind 图标 (方法论/经验)
- hover 显示 1 行摘要

### 9.4 IngestConfirmCard 组件

AI 建议沉淀时渲染:
- 预览卡片 (title + insight + tags + insight_type)
- 确认按钮 → 调用 atomsApi.create()
- 取消按钮 → 关闭卡片

## 10 · 输入区域设计

### 10.1 布局


### 10.2 附件功能

- 左下角 📎 按钮
- 支持图片 (jpg/png/gif/webp) + 文本文件 (.txt/.md/.json)
- 图片: base64 编码后作为 multimodal content part 发送
- 文件: 读取文本内容注入消息
- 预览条: 已选文件的缩略图/图标，可删除

### 10.3 / 命令选择器

输入 `/` 时弹出 skill 选择面板:
- `/read` → 强制触发知识检索模式
- `/write` → 进入沉淀引导模式
- `/mentor` → 触发认知复盘
- `/model` → 切换当前会话模型

### 10.4 模型选择器

- 输入框右下角小标签显示当前模型名
- 点击弹出下拉: 列出所有 enabled 的 LLM 模型
- 默认 "auto" (使用设置中的默认模型)
- 切换后仅影响当前会话
