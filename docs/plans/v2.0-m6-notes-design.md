## 0 · 为什么做笔记

### 北极星对齐

Atomsyn 的核心使命是**让 AI 越来越懂用户**。当前的认知沉淀路径是:

```
用户与 AI 对话 → atomsyn-write 沉淀碎片 → atomsyn-read 唤醒 → 方法论相遇
```

但这条路径有一个**断裂点**: 用户的日常思考、阅读笔记、会议记录等**非 AI 对话场景**产生的认知,没有入口进入 Atomsyn。

笔记模块补上这个断裂点:

```
用户日常笔记 (M6)
     ↓ 一键沉淀
atomsyn 碎片 → 自动关联方法论 → AI 唤醒
     ↑
用户与 AI 对话 (现有)
```

### 不做什么 (M6 范围边界)

- ❌ 不做云同步 (本地优先,不变)
- ❌ 不做协作/分享
- ❌ 不做插件系统
- ❌ 不做 Obsidian 级别的双链/图谱
- ❌ 不做真实加密 (延后)
- ❌ 不做 AI 自动整理笔记
- ❌ 不做智能分组 (用户主权 — 纯用户文件夹控制)
- ❌ 不做 SQLite 索引 (bigram 内存搜索足够)

## 1 · 产品定位

**一句话**: 一个轻量的、桌面级体验的 Markdown 笔记模块,嵌入 Atomsyn 作为认知沉淀的**上游入口**。

**与 Bear/Obsidian 的区别**: 笔记内容可一键成为 atomsyn 碎片,与方法论骨架相遇。Bear 的笔记是孤立的;Atomsyn 的笔记是有结构的认知资产的一部分。

## 2 · 用户场景

### 场景 A: 日常记录
> 用户读了一篇文章,快速打开 Atomsyn 笔记,用 Markdown 记下要点和感想。
> 附上截图。保存后这条笔记就在本地,随时可以回顾。

### 场景 B: 会议记录 → 沉淀
> 用户开完产品评审会,在笔记中记录决策和讨论要点。
> 点击工具栏"沉淀到 Atomsyn" → LLM 解析全文 → popover 预览碎片 → 确认沉淀。

### 场景 C: 精准段落沉淀
> 回顾笔记时,选中某段有价值的内容 → 右键 → "沉淀选中内容" → LLM 解析选中段落 → 生成碎片。

### 场景 D: 从碎片追溯笔记
> 用户在方法论详情页看到 📎 碎片,点击"查看原始笔记" → 跳转到笔记编辑器。

## 3 · 数据架构

### 3.1 存储位置 (物理文件夹映射)

```
~/Library/Application Support/atomsyn/notes/
├── meta.json                         ← 全局元数据 (分组树、排序)
├── <group-folder>/                   ← 用户自定义分组 (物理文件夹)
│   └── <note-slug>/                  ← 每个笔记一个文件夹 (slugified title)
│       ├── meta.json                 ← 笔记元信息
│       ├── content.md                ← 笔记正文 (纯 Markdown)
│       └── attachments/              ← 附件目录
│           ├── screenshot-1.png
│           └── diagram.pdf
└── .trash/                           ← 废纸篓 (软删除的笔记)
    └── <note-slug>/
```

### 3.2 笔记元信息 (per-note meta.json)

```jsonc
{
  "id": "note_<slug>_<timestamp>",
  "title": "产品评审会纪要 2026-04-10",
  "tags": ["会议", "产品决策"],
  "pinned": false,
  "crystallizeStatus": "none",       // "none" | "parsing" | "parsed" | "failed"
  "crystallizedAt": null,            // ISO timestamp
  "linkedFragments": [],             // 沉淀出的碎片 ID
  "groupId": "工作-产品",             // 对应物理文件夹
  "wordCount": 342,
  "createdAt": "2026-04-10T14:30:00+08:00",
  "updatedAt": "2026-04-10T15:45:00+08:00",
  "deletedAt": null                  // 非空 = 在废纸篓中
}
```

### 3.3 笔记正文 (content.md)

纯 Markdown。媒体通过相对路径引用:
```markdown
# 产品评审会纪要

## 决策一: 放弃实时协作

团队一致同意聚焦个人用户...

![评审会白板照片](./attachments/screenshot-1.png)
```

### 3.4 全局元数据 (notes/meta.json)

```jsonc
{
  "version": 1,
  "groups": [
    { "id": "工作-产品", "name": "产品", "parentId": "工作", "sortOrder": 0, "createdAt": "...", "updatedAt": "..." },
    { "id": "工作-工程", "name": "工程", "parentId": "工作", "sortOrder": 1, "createdAt": "...", "updatedAt": "..." },
    { "id": "工作", "name": "工作", "parentId": null, "sortOrder": 0, "createdAt": "...", "updatedAt": "..." },
    { "id": "阅读笔记", "name": "阅读笔记", "parentId": null, "sortOrder": 1, "createdAt": "...", "updatedAt": "..." }
  ],
  "defaultGroup": "工作-产品",
  "sortOrder": "updatedAt"
}
```

### 3.5 标题规则

- 默认从 content.md 首行 `# ` 提取
- 若用户在 meta.json 中自定义了 title,则 meta 优先
- note 文件夹名使用 slugified title (中文保留)

### 3.6 解析状态 (crystallizeStatus)

```
none → parsing → parsed
         ↓
       failed
```

- 内容修改后保持 `parsed`,UI 显示"有新修改"提示
- 用户可再次触发沉淀,追加新碎片到 linkedFragments

```
  可折叠为     可拖拽调宽         flex 填充剩余

```

| 面板 | 默认 | 最小 | 最大 | 行为 |
|------|------|------|------|------|
| GlobalSidebar | 240px | 48px (图标) | 240px | 拖拽/双击切换折叠 |
| 笔记列表 | 280px | 200px | 400px | 拖拽调宽 |
| 编辑器 | flex-1 | 400px | — | 填满剩余 |

**图标模式 (48px)**: Logo + 模式图标列 + 底部操作，展开后恢复完整树。

### 4.2 侧边栏 (NotesSidebar)

**无智能分组**。用户完全自控:

1. **"新建笔记" CTA** (amber 渐变)
2. **📁 全部笔记** (固定入口，count badge)
3. **用户文件夹树** (递归，支持层级)
   - 展开/折叠 chevron
   - 右键: 新建子文件夹 / 重命名 / 删除
   - 拖拽排序
   - 接受笔记拖入 (跨面板 DnD)
4. **🗑 废纸篓** (固定底部，count badge)

### 4.3 笔记列表

每条笔记:
- 标题 (Inter Medium, truncate)
- 正文前 2 行预览 (跳过 `#` 行)
- 相对日期 + 分组路径
- Tags chips (max 3)
- Pin 指示器
- CrystallizeStatus 指示器

支持:
- 搜索 (bigram, 标题+正文)
- 排序 (updatedAt / createdAt / title)
- 置顶笔记始终在最上方

### 4.4 Markdown 编辑器

**纯 WYSIWYG** (Bear 风格),基于 **TipTap** + `tiptap-markdown`:
- 所见即所得,不暴露 Markdown 源码
- 导出/复制到其他 Markdown 工具保持格式一致
- 支持: 标题 / 粗体 / 斜体 / 列表 / 有序列表 / 待办 / 引用 / 代码块 / 分割线 / 链接 / 表格
- 图片: 粘贴/拖拽 → 自动保存到 attachments/ → 编辑器内渲染
- **自动保存**: 1.5s debounce,切换笔记时同步 flush,新建即创建空文件

### 4.5 交互细节

| 操作 | 行为 |
|---|---|
| 新建笔记 | Cmd+N 或侧边栏 "+" 按钮 |
| 删除笔记 | 右键 → 删除 → 移到废纸篓 |
| 彻底删除 | 废纸篓中右键 → 永久删除 |
| 移动笔记 | 拖拽到分组 / 右键 "移动到..." |
| 标签管理 | 编辑器头部 tag 区域 |
| 置顶/取消 | 右键菜单 |
| 搜索 | 列表顶部搜索栏 |

## 5 · 与认知层的集成

### 5.1 笔记 → 碎片 (沉淀)

**两个入口**:

| 入口 | 范围 | 触发 |
|------|------|------|
| 工具栏 "沉淀到 Atomsyn" 按钮 | **全文** | 点击按钮 |
| 右键菜单 "沉淀选中内容" | **选中段落** | 选中文字后右键 |

**流程**:
1. 获取内容 (全文或选中)
2. 调用 LLM (crystallize.md prompt) 解析
3. LLM 返回 1-5 个碎片候选 (自动决定拆分粒度)
4. 弹出**轻量 popover** (非 dialog) 显示预览
   - 每个碎片: checkbox + title + insight_type chip + tags
   - 用户可取消勾选不想沉淀的
5. 确认 → 批量创建碎片 (atomsApi.create)
6. 更新笔记 meta: `crystallizeStatus='parsed'`, `crystallizedAt`, `linkedFragments[]`
7. 碎片 context: `source='note'`, `noteId=xxx`

**不做快捷键** (避免误操作)。

### 5.2 碎片 → 笔记 (追溯)

碎片详情页/📎 面板: 如果碎片 `context.source === 'note'`,显示"查看原始笔记"按钮 → 跳转。

### 5.3 AI 静默发现 (V2.x 远期,M6 不实现)

预留 `crystallizeStatus` 和 `linkedFragments` 字段作为接口。

Extensions: StarterKit, TaskList/TaskItem, Link, Table, Image, Placeholder, HorizontalRule, CodeBlockLowlight

| Method | Path | 行为 |
|--------|------|------|
| GET | /api/notes/meta | 全局元数据 |
| PUT | /api/notes/meta | 更新全局元数据 |

| GET | /api/notes/:noteId | meta + content |
| POST | /api/notes | 创建笔记 |
| PUT | /api/notes/:noteId | 更新笔记 |
| DELETE | /api/notes/:noteId | 软删除 |
| POST | /api/notes/:noteId/move | 移动 |
| POST | /api/notes/:noteId/restore | 恢复 |
| DELETE | /api/notes/trash/:noteId | 永久删除 |
| GET | /api/notes/trash | 列出废纸篓 |
| POST | /api/notes/:noteId/attachment | 上传附件 |

### 6.3 路由

```
/notes              → 笔记主页 (三栏布局)
/notes/:noteId      → 打开特定笔记
```

### 6.4 搜索

Bigram 内存搜索,复用 CLI 已有的 `tokenizeQuery()` 思路。不上 SQLite。

## 8 · M6-Pre 踩坑记录 (复用)

### 路径解析
`openContainingFolder(absPath)` 需绝对路径。笔记在 `notes/` 下,需同样的 dev/Tauri 路径拼接逻辑。

### 右键菜单事件时序
`requestAnimationFrame` 延迟注册监听器,避免同帧触发关闭。`createPortal(document.body)` 避免 sidebar overflow 裁剪。

### 阻止浏览器默认右键菜单
侧边栏根容器 `onContextMenu={e => e.preventDefault()}`。

## 9 · 已决策 (2026-04-10 锁定)

| 决策项 | 结论 |
|--------|------|
| 智能分组 | **不做**,纯用户文件夹 + 拖拽管理 |
| 编辑器模式 | **纯 WYSIWYG** (Bear 风格),导出标准 Markdown |
| 存储方案 | **物理文件夹映射** (方案 A) |
| 文件夹命名 | **slugified title** (中文保留) |
| 三栏拖拽 | **全部可拖拽**,左侧可折叠为图标 |
| 沉淀触发 | **工具栏(全文) + 右键(选中)**,不做快捷键 |
| 沉淀粒度 | **一对多**,LLM 决定碎片数 |
| 沉淀反馈 | **轻量 popover** 带 checkbox 预览 |
| 标题来源 | meta.title 仅通过显式重命名设置; title 为空时列表显示 content 首行前 40 字符 |
| 自动保存 | **1.5s debounce**,新建即创文件,切换时 flush |
| 置顶 | **保留**,右键菜单设置 |
| 解析状态 | **parsed 保持**,修改后提示"有新修改" |
| 搜索 | **bigram 内存**,不上 SQLite |
| "全部笔记" | **保留**为固定入口 |
| "废纸篓" | **保留**为固定底部入口 |

### 已知限制

- **Markdown input rules + CJK**: `# ` 转标题和 `**text**` 转粗体需要用英文半角输入。中文 IME 的 composition 事件会打断 TipTap 的 input rule 匹配。主要使用工具栏按钮或 Cmd+B/I。
- **TipTap StarterKit 已内置 heading/bold/italic input rules**: 半角 `#` + 空格 = H1,`##` + 空格 = H2,`**text**` = 粗体,`*text*` = 斜体。

### ✅ Sprint 5 — 笔记提炼 (Note → Fragment) (完成)

> 目标: 实现笔记到经验碎片的提炼闭环 (核心差异化功能)

#### T5.1 · Crystallize Prompt [S]
**新建** `scripts/ingest/prompts/crystallize.md`

基于 `classify.md` 结构:（这里需要说明一个潜在的问题scripts/ingest/prompts/classify.md 的提示词解析逻辑存在局限，提示词中明显写到了按照哪些分类进行经验提取，但是这是过拟合的做法，我们需要参考skills/atomsyn-write/SKILL.md skill中的做法来实现泛化性，基于当前用户实际已有的分类，避免语义相同但名字不同的分类重复出现，同时也避免了分类过拟合的问题，skill是给Agent使用我们cli时使用的，而classify.md和笔记沉淀功能是用户主动操作时触发LLM解析的，因此我们需要保证两边的逻辑处理方式逻辑和内核是一样的）
另外需要说明一点，原本`classify.md` 是服务于GUI端用户主动进行'沉淀资产'操作时触发使用到的提示词，但是按照我们最新的设计和架构，GUI端所有的沉淀动作其实都迁移到了这次笔记沉淀中了，所以本质上本次的提示词就是替换`classify.md` 的，通过这里的笔记沉淀功能来实现经验的洞察提取；而GUI端'聊天'对话中不再实现'沉淀资产、沉淀一条经验'这个动作，而仅作为基于个人资产知识库分析、推荐方法论、复盘等操作；这是笔记大的变化；
```
输入: 笔记内容 (全文或选中段落)
输出: JSON 数组,1-5 个碎片
每个碎片: { title, insight, sourceContext, role, situation, activity, insight_type, tags, confidence }
规则:
- 提取有价值的经验洞察,不是事实记录
- 每个碎片独立成文
- 如果纯事实记录无经验洞察,返回空数组
```
