# Atomsyn V2.2 · Bootstrap (引导式批量冷启动)

> **版本**: V2.2 (proposed, 实施中)
> **状态**: F 组文档同步 (本文档归属)
> **上游**: `docs/framing/v2.x-north-star.md` · `docs/plans/v2.x-vision-handoff.md` §9.5 (cognitive-evolution 已合并) · `openspec/changes/2026-04-bootstrap-skill/`
> **目的**: 让任意会话不读 spec 也能理解"这次为什么做 bootstrap、怎么落地、与 cognitive-evolution 怎么联动"

---

## 1 · 一句话叙事

V2.2 通过新增第 4 个 Skill `atomsyn-bootstrap` + CLI 子命令 `atomsyn-cli bootstrap`, 让用户**第一次唤醒**自己积累多年的认知资产 —— 把硬盘上散落的 markdown / 笔记 / 历史聊天导出, 按 5 层架构 (Profile / Preferences / Episodic / Domain / Reflections) 提炼成 1 条 profile atom + N 条 experience/fragment atom, 一次性装进仓库。

---

## 2 · 北极星对齐

V2.x 北极星 (`docs/framing/v2.x-north-star.md`) "**让你积累的认知, 在需要时醒来**" 在仓库层有一个隐含前提 —— **必须先有"沉睡之所"到"醒来"的桥**。V2.0 M0-M6 兑现了"存得住、找得到、AI 能调", V2.1 P0+P1 兑现了"看见 + 推动复盘", cognitive-evolution (V2.x P0, 2026-04 已合并) 兑现了"醒来时还是新鲜的"。但所有这些都假设用户从"积累过程"开始, 一次记一条 —— 对全新用户来说, 知识地图全空, atomsyn-read 永远沉默, atomsyn-mentor 永远说"数据不足", 北极星 Demo 场景里 "两个月前的顿悟会安静地出现在回答里" 无法兑现。

bootstrap 落在**仓库层**的"第一次唤醒": 让用户已经积累但散落、沉睡在硬盘各处的认知, **第一次进入 Atomsyn**, 同时喂养下游的结构层 (profile 把碎片串成画像骨架) 和教练层 (未来 mentor 可读 profile 做 declared vs inferred gap 分析, v2)。三层互相喂养的回路从此真正闭合。

## 3 · 用户故事 (北极星 Demo 扩展)

> 全新用户 A 装上 Atomsyn, 打开 GUI 进聊天页, 看到一个明显的"初始化向导"按钮。点击 → 选择 `~/Documents` → Phase 1 TRIAGE 出概览 (800 个 .md, 12 MB, 跨 18 个月) → 用户确认范围。
>
> Phase 2 SAMPLING 抽样 15 个文件 → 系统给出"你看起来主要是前端工程师 + 独立产品开发者, 主要在 Rust/Tauri + AI Agent 领域, scope_appetite 0.7 / risk_tolerance 0.4 / detail_preference 0.8 / autonomy 0.5 / architecture_care 0.85" → 用户校准。
>
> Phase 3 DEEP DIVE dry-run → 输出 markdown 报告 (50 条候选 atom + 1 条 profile 草案), 持久化到 session `.md` 文件 → 用户在 markdown 上删除 5 条不准确的候选, 改了 3 条 name → 点"确认写入"。
>
> commit 阶段 → LLM 把保留的 markdown 候选生成完整 atom JSON → 通过 `atomsyn-cli ingest` 写入 → reindex。用户去新建的 ProfilePage → 看到 5 维滑块 + 时间线 (v1 trigger=`bootstrap_initial`) → 微调 → 切 `verified=true`。
>
> 用户回到 Atlas → 看到 50 条新 atom 进入双骨架 → 在 Cursor 开新会话提一个相关问题 → atomsyn-read 命中 atom (lastAccessedAt 自动更新)。
>
> 一个月后, 用户跑 atomsyn-mentor → 看到 staleness 信号 + 主动建议 prune → 用户裁决 → archive 一些旧 import。
>
> 三个月后, GUI 提示"画像 90+ 天未校准" → 用户回 ProfilePage 校准 → 旧 profile 入 `previous_versions[]`, 新数据生效。
>
> **存 → 看见 → 行动 → 相遇。这就是 V2.x 的认知复利, bootstrap 是它第一次启动的钥匙。**

详细 Demo 文本见 `docs/framing/v2.x-north-star.md` §4.4。

## 5 · 与 cognitive-evolution 联动

bootstrap-skill 是 cognitive-evolution 的天然下游, 复用其多个核心接口 (无重复实现):

| 共享接口 / 字段 | 来源 (cognitive-evolution) | bootstrap-skill 如何用 |
|---|---|---|
| `applyProfileEvolution(deps, {newSnapshot, trigger, evidenceDelta?})` | `scripts/lib/evolution.mjs` | bootstrap **B13 commit 阶段** 写入 profile 时调用 (trigger=`bootstrap_initial` / `bootstrap_rerun`); GUI ProfilePage **校准模块** 提交时 (trigger=`user_calibration`); restore 历史时 (trigger=`restore_previous`) |
| `computeStaleness(atom, now)` 含 profile_factor | 同上 | profile 90 天未校准 → factor=1.5x → GUI staleness 提示自动生效 |
| imported atom fallback (createdAt 兜底) | 同上 (computeStaleness 内) | bootstrap 写入的 atom 默认 `confidence=0.5` + `lastAccessedAt=null` 时, staleness 用 createdAt 兜底, 不会"刚 import 立即被标 stale" |
| 演化字段 mixin (lastAccessedAt / archivedAt 等) | `skills/schemas/atom.schema.json` 5 字段 + TS `AtomEvolutionFields` | profile-atom.schema.json 共享 lastAccessedAt / archivedAt / archivedReason; supersededBy/supersedes 在 profile 上**不使用** (D-008, profile 用 `previous_versions[]` 替代 supersede 链) |
| 5 个 cognitive-evolution API 端点 (calibrate / supersede / archive / restore / track-view) | `vite-plugin-data-api.ts` + `src/lib/tauri-api/routes/atoms.ts` | bootstrap 不直接消费, 但**新增**的 8 个端点 (4 profile + 4 bootstrap session) 模仿这套双通道模式 |
| `src/lib/atomEvolution.ts` (TS port) | 同名文件 | bootstrap GUI 端 (BootstrapWizard / ProfilePage) 计算 profile staleness 时直接 import |

**核心不变量** (两个 change 共同维护):
- D-002 不删除文件, 只 archive (软删除)
- D-005 prune 永远 dry-run, LLM 严禁自动 mutate 知识库
- D-008 profile 是单例, 不进 supersede 关系网, 用 `previous_versions[]` 演化
- D-010 (本 change) profile id 固定 `atom_profile_main`, 单例 + previous_versions[] 数组追溯历史
- D-011 (本 change) bootstrap 写入永不一步到位, 必须经过 dry-run + commit 两阶段
- D-012 (本 change) bootstrap LLM prompt 模板 hard-code 在 `scripts/bootstrap/prompts/`, 不放配置文件

## 6 · V1 仅观察哲学 (D-007 / D-013 联动)

bootstrap v1 必须遵守"profile 仅观察"哲学 —— 这是 plan-tune 的成熟参考做法, 也是用户主权的体现:

- **profile.verified 默认 false**: bootstrap commit 阶段写入的 profile 不被 atomsyn-read 自动注入到 system prompt (R-I6 / B-I3)
- **校准入口在 GUI** (D-013): 用户必须在 GUI ProfilePage 走完一次校准流程 (调整 5 维滑块 + identity / domains / patterns 文本框) 才能切 `verified=true`; 切到 true 时调 `POST /atoms/:id/calibrate-profile`, 旧版本被推入 `previous_versions[]` (trigger=`user_calibration`)
- **Skill v1 不消费 profile**: atomsyn-read / atomsyn-mentor 在 v1 完全不动 profile 数据路径; v2 才考虑放开 read 注入和 mentor gap 分析
- **错误画像不会无声扩散**: 一个 LLM 推断错误的 profile 不会因为"用户没及时校准"就污染所有未来对话; verified=false 状态下, profile 是"产出但被审视前的草稿"

没有 GUI 校准入口的话 v1 仅观察哲学走不通 (verified 永不变 true 等于 read 永远不能用 profile 等于功能死掉), 这就是 D-013 把 GUI 校准模块定为 v1 必交付的根本原因。
