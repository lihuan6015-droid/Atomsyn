## 💡 Idea #001 · "FOMO 缓解器" 信息渠道聚合

**沉淀日期**: 2026-04-08
**提出场景**: 新技能/新知识爆炸时代，用户对"我是不是漏掉了重要信息"的焦虑日益严重

### 背景
我们的核心痛点是「学过即忘」，但其实有一个**前置痛点**：在还没学之前，**用户根本不知道自己应该学什么**。每天朋友圈/Twitter/Hacker News/微信公众号/小红书里飞过去几十条"重要"内容，看完忘了，没看又焦虑，形成"信息瀑布 → FOMO → 焦虑 → 假装学习"的恶性循环。

### 设想
在 CCL PM Tool 内部加一个 **"信息渠道" / Feed Hub**：
- 用户在设置中订阅自己感兴趣的信息源（RSS / Hacker News / Product Hunt / 微信公众号 / 知乎专栏 / Twitter list / Substack ...）
- 工具定时（每天 1-2 次）拉取最新内容
- 通过配置好的 LLM 对每条内容做**智能筛选**：
  - 这条内容是否真的对"我"有用（基于用户已有知识库 + 当前在做的项目）
  - 如果有用，归属到哪个骨架 / 哪个阶段 / 哪些已存在的方法论卡片
  - 自动生成一段精简摘要 + 提取关键概念
- UI 上是一个**每日 5 分钟 Feed**：上滑卡片式，每张卡片显示来源 + 摘要 + "归档为知识原子" 按钮

### 需要澄清
- 信息源接入方式：纯 RSS（最简）/ 自建抓取脚本 / 第三方聚合 API（如 RSSHub）
- LLM 筛选成本：每天几十条 × 每条 ~500 token = 单日成本 ≈ ?
- 用户 LLM 配置应该和 Copilot 共享一个，还是单独配置？
- 隐私边界：朋友圈、微信收藏类源是否要碰？（建议不碰）
- 是否要把"今日 Feed"作为一级 Tab，还是嵌在 🌱 成长档案 里？

### 风险
- 🔴 **悖论风险**：FOMO 工具本身会变成新的 FOMO 来源——必须严格限制每日 Feed 数量（≤10 条），并强制用户"选择性消费而非穷尽消费"
- 🟡 LLM 成本可能不可控
- 🟡 RSS / 抓取的可靠性问题（很多源不提供 feed）

### 初步评估
- **价值**: ⭐⭐⭐⭐ 高（直击当代知识工作者最普遍的隐性痛点）
- **复杂度**: 🔧🔧🔧🔧 高（涉及数据接入 + 后台调度 + LLM 流式处理）
- **建议层级**: **v2.0**（先让 v1 的"沉淀"半身先跑稳，再加"流入"半身）

## 💡 Idea #002 · "活的浏览器收藏夹" — 智能网页归档

**沉淀日期**: 2026-04-08
**提出场景**: 浏览器收藏夹永远是静态、不可搜、不可复用的死数据

### 背景
所有人都在用浏览器收藏夹，但所有人都同时承认收藏夹**等于墓地**——加进去那一刻就再也不会回来。它的核心问题：
1. 只存 URL，不存内容（页面挂了/改了你不知道）
2. 没有摘要，几个月后看标题完全想不起当时为什么收藏
3. 不能搜索内容（只能搜标题）
4. 没有标签 / 分类语义
5. 跟"我现在在做的项目"完全没关系

### 设想
在 CCL PM Tool 里做一个 **🔖 智能书签** 模块（可能作为知识图书馆里的一种特殊"骨架"，或独立 Tab）：
- 用户通过浏览器扩展（或者直接粘贴 URL）添加一个网页
- 工具调用配置好的**多模态 LLM**：
  - 抓取页面 HTML（或者把页面截图传给视觉模型）
  - 生成一段结构化摘要：是什么 / 解决什么问题 / 关键要点（3-5 条）/ 潜在用途
  - 自动生成 3-6 个标签
  - 自动归属到现有骨架的某个 cell（如果适用）
  - 拍一张干净的页面截图作为视觉锚点
- 卡片视觉：和方法论原子卡用同样的渐进披露 UI，但增加 "📸 页面截图" 区
- 与项目演练场打通：在某个项目下也能挂书签，作为"参考资料"
- 与 Copilot 打通：Copilot 在推荐方法论时也能推荐相关书签
- **与"个人收藏夹"字段打通**：原子卡里的 bookmarks 字段未来从纯文本/链接升级为带摘要+截图的智能书签

### 需要澄清
- 抓取方式：
  - 选项 A：用户手动复制粘贴页面 URL，工具自己 fetch（受 CORS 限制）
  - 选项 B：浏览器扩展配合（"右键 → 添加到 CCL PM"）
  - 选项 C：截图上传（用户用系统截图工具截好再丢进来）
- 多模态 LLM 选型：Claude 3.5 Sonnet、GPT-4V、Qwen-VL ……是否要在设置里允许配置一个独立的 vision provider？
- 截图存储：放本地 `data/bookmarks/<id>.png`？需要担心 git 仓库膨胀
- 失效检测：定期检查 URL 是否还活着，挂掉时显示"⚠️ 页面已失效，但有截图存档"

### 风险
- 🟡 多模态 LLM 调用成本（截图理解比纯文本贵 5-10×）
- 🟡 图片存储膨胀（可考虑只存第一屏 + 压缩）
- 🟡 部分网站防爬（403 / Cloudflare 拦截）
- 🟢 整体技术风险可控

### 初步评估
- **价值**: ⭐⭐⭐⭐⭐ 极高（这个痛点几乎是普世的）
- **复杂度**: 🔧🔧🔧 中（核心是接 vision API + 一个新的卡片样式）
- **建议层级**: **v1.5**——这是 v1 已有"个人收藏夹"字段的天然升级，路径清晰
- **优先级**: 在所有 ideas 里我个人最看好这一条

## 💡 Idea #003 · 本地 AI Skill 感知与解读

**沉淀日期**: 2026-04-08
**提出场景**: 用户装了一堆 Claude / Codex / Trae 的 skill 但想不起来什么时候用

### 背景
作者本人就是个典型例子——`/Users/circlelee/.claude/skills/` 里有 100+ 个 skill，每次想起来"诶我之前装过一个相关的"但是叫什么名字、长什么样、什么时候触发，全都忘了。这和我们核心的「学过即忘」痛点本质完全一样，只是对象从"方法论"变成了"AI Skill"。

### 设想
作为知识图书馆里的一种**新骨架**："🤖 AI Skill 库"：
- 工具自动扫描以下目录（用户可在设置里增加路径）：
  - `~/.claude/skills/`
  - `~/.codex/...`
  - `~/.trae/.../skills/`
  - 任何用户指定的 plugins 目录
- 对每个 skill 的 `SKILL.md` 头文件（YAML frontmatter + description）做解读
- 通过配置的 LLM 自动生成：
  - **核心理念**（一句话概括 skill 的定位）
  - **典型场景**（什么时候应该想到它）
  - **触发关键词**（用户说什么话时这个 skill 该被调出来）
  - **配套 prompt 示例**（一段可以投入使用的对话开头）
  - **相关 skill**（同类型其他 skill 的链接）
- 每个 skill 在我们的 UI 里就是一张原子卡片，结构和方法论原子完全一样
- **特色功能**：在 Atlas 矩阵视图里，按"用户场景维度"重新组织，比如"产品需求分析" / "代码 review" / "文档生成"……让 100+ skill 不再是字母排序的列表
- 自动检测：当用户的 skill 目录新增/删除文件时，自动同步更新

### 需要澄清
- 文件系统监听：v1 的 Vite dev plugin 模式下要怎么监听 `~/.claude/skills/`？需要把它们 symlink 到 data 目录吗？还是用 chokidar？
- 隐私边界：要不要扫描 skill 文件的完整内容（可能含敏感 prompt）还是只看 frontmatter？
- 解读成本：100 个 skill × 解读一次 ≈ 一次性 ¥1-3，可以接受
- 跨工具支持：Claude / Codex / Trae 的 skill schema 不同，需要适配器
- 与方法论卡片的关系：是同一种数据结构（atom）还是新的实体类型（skill）？
  - 倾向于：用 atom 复用，加一个 `kind: 'skill' | 'methodology'` 字段区分

### 风险
- 🟡 文件系统访问权限（v1 Web 模式下没有，必须等 Tauri 包壳）→ 这点强烈推荐 v2 在 Tauri 上做
- 🟢 LLM 解读已成熟
- 🟢 数据结构改动小

### 初步评估
- **价值**: ⭐⭐⭐⭐ 高（对深度 AI 工具用户极具价值，但用户群相对窄）
- **复杂度**: 🔧🔧🔧 中（核心是文件扫描 + 适配器 + LLM 调用）
- **建议层级**: **v2.0**（最好等 Tauri 上线后做，文件访问问题自然解决）
- **强依赖**: Tauri 桌面壳 + Idea #002 的视觉/交互模式可以直接复用

## 💡 Idea #004 · UI/UX 骨架内容 + 视觉资源收藏夹

**沉淀日期**: 2026-04-08
**提出场景**: 作者在和 AI 共创前端时只能说"Apple 风 / Google 风"，没有专业术语，无法精确表达，每次都要从零摸索

### 背景
作者明确说过的痛点：「我和你一起开发前端时，我们一起磨合出很多精美的页面，但是我不知道那些页面视觉上的专业名词或功能按钮的叫法，或者动画专业称呼，以至于到未来我再次用到时，我永远的只能和你说 Apple 风、Google 风」。

这是一个**专业词汇缺失**的问题——不是不会用，而是不会"指"。

### 设想
在 v1 已经预留的 **UI/UX 骨架**下补全实质内容（v1 仅占位）：

#### Part A · 视觉设计语言知识体系
做一组 cell + 原子，覆盖主流设计语言：
- **Apple HIG（Liquid Glass / Glassmorphism / SF Pro）**
- **Google Material 3（Dynamic Color / Elevation / Motion）**
- **Linear / Raycast / Vercel 风（Modern Minimal）**
- **Brutalism / Neobrutalism**
- **Skeuomorphism Revival**
- **Ant Design / Fluent UI / Carbon**

每个原子的 `aiSkillPrompt` 字段是一段可以直接投入"AI 设计副驾驶"对话的 prompt，例如：
> "你是一位精通 Apple HIG 的资深设计师……我要做一个 [组件类型]，请给我一份符合 Apple Liquid Glass 风格的视觉规范（圆角 / 阴影 / 颜色 / 动画 / 字体），并附 React + TailwindCSS 实现示例……"

#### Part B · 视觉资源收藏夹（专属卡片类型）
为某些 cell 加入一种新的子类型——**资源类原子**：
- React 组件库网站（shadcn/ui · Aceternity UI · Magic UI · ReactBits · Origin UI ……）
- 图标库（Lucide · Heroicons · Phosphor · Tabler · Iconify ……）
- 字体库（Google Fonts · Adobe Fonts · Vercel Fonts ……）
- 配色工具（Coolors · Realtime Colors · Tailwind Colors）
- 动画库（Framer Motion · GSAP · Lottie · React Spring ……）
- 灵感站点（Mobbin · Dribbble · Page Flows · Httpster ……）

每张资源卡片含：
- 网站 logo + 截图
- 一句话定位
- 我个人用过哪些组件 / 哪些坑（可日后追加）
- 该资源最适用的"骨架阶段"（比如 Mobbin 适合 Discover 阶段做竞品研究）
- **跨向链接**：可以从产品创新 24 步法的 cell 14（体验地图）反向链到这里

#### Part C · 个人项目沉淀回流
当作者在某个项目里"我和 Claude 一起磨合出某个组件"时，可以一键把它**反向沉淀**为一张"我的视觉沉淀"原子，包含：
- 截图
- 当时和 AI 的对话片段
- 最终的 React + Tailwind 代码片段
- "这个组件叫什么"——AI 帮你贴标签（"是 Glassmorphism Card with Soft Shadow"）

下次再做类似组件，直接到这张卡片：复制 prompt → 用同一份术语跟 AI 沟通 → 三秒复刻。

### 需要澄清
- 跨骨架的卡片类型如何区分？（建议：原子的 `kind` 字段加 `'resource' | 'methodology' | 'visual-spec'` 三种）
- 截图怎么存（同 Idea #002）
- 资源类原子的 `aiSkillPrompt` 字段可以为空，但保留"打开网站"按钮？
- Part C 的"反向沉淀"是否可以和 Claude Code 协作？通过一个新的 ingest skill `ingest-visual.skill.md`

### 风险
- 🟢 Part A（设计语言原子）→ 可以直接调用 6 列骨架的 atom 沉淀流程，零技术风险
- 🟡 Part B（资源卡片）→ 需要新的卡片类型 + 截图存储
- 🔴 Part C（项目反向沉淀）→ 需要复杂的 AI 对话回放 + 代码片段提取

### 初步评估
- **价值**: ⭐⭐⭐⭐⭐ 极高（直接对应 PM 转型 + 跨界并行成长的核心场景）
- **复杂度**: 分级
  - Part A：🔧 低（纯内容补全，无代码改动）
  - Part B：🔧🔧 中（新卡片类型 + 截图）
  - Part C：🔧🔧🔧🔧 高
- **建议层级**:
  - Part A → **v1.1**（可以马上让 sub-agents 写一波 UI/UX 设计语言原子，类似 24 步法那样）
  - Part B → **v1.5**
  - Part C → **v2.0**

## 📋 后续评估流程

每个想法在评估时回答这 4 个问题：

1. **Pull**：v1 真实使用中，这个痛点有没有真的咬到我？
2. **Push**：实现这个想法需要的最小可行版本是什么？是否能砍到 v1.5 而不是 v2？
3. **Trade-off**：实现它会牺牲哪些已有功能的演进时间？
4. **Verdict**：投入 / 推迟 / 永远不做
