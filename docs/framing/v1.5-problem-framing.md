## Phase 1 · Look Inward (向内看)

### 1.1 问题症状（What is the problem?）

**个人记忆资产无法被自己和 AI 共同高效检索调用。**

表层有三个切面，本质是同一个根问题：

1. **学过即忘的方法论焦虑**（V1 PRD 的原始定位）—— 学了一堆产品/UI/Agent 方法论，几个月后只剩名词，真到工作场景调不出来
2. **资产坟墓化** —— 浏览器收藏夹、本地素材、装过的 skills，加进去那一刻就再也不会回来
3. **AI 时代的认知错配** —— 个人沉淀是"死的"，AI agent 用不了；每次和 AI 协作都要从零喂上下文，自己积累的资产对 agent 完全透明

> ⚠️ **关键判断**：V1 PRD 只解决了第 1 切面，把它当成了完整问题。新定位下，必须把三个切面作为同一个问题处理。

---

### 1.2 为什么至今没解决？（Why haven't we solved it?）

| # | 障碍 | 类型 |
|---|---|---|
| 1 | 范式太新（Skills/MCP 协议刚成熟，2025 年之前根本做不出来） | 时间窗口 |
| 2 | 技术太难（跨类型资产 + 本地优先 + 视觉 + Agent 接口，缺一不可） | 技术复合性 |
| 3 | 商业模型不成立（大厂没动力，小团队不划算） | 经济结构 |
| 4 | 思维定势（行业把"知识管理"和"AI 接口"当成两件事） | 认知盲区 |
| 5 | 用户群被忽略（中段 AI 用户人数少、无市场声量） | 群体可见性 |

> 💡 这五条加起来构成一个画像：**这是一个"窗口期 + 结构性小生境"的问题**——技术刚成熟、行业还没反应、目标用户还在边缘、但需求是真实的。

---

### 1.3 我自己是问题的一部分吗？（How are we part of the problem?）

**5 条已识别的偏见**：

1. **范围蠕动倾向** —— 天然喜欢"系统化、全覆盖"的设计，可能把可以小切口验证的问题硬撑成"个人记忆资产平台"
2. **工具完美主义** —— 倾向"先把工具打磨到自己满意才用"，可能让工具本身成为目的
3. **解决方案先行** —— 已经在脑子里有了"4 个 idea + 统一 skill 接口"的解法，可能在没充分理解问题前就锁死了答案空间
4. **自我样本偏见** —— 我是极端样本（跨界 + 深度 AI 用户 + 前端能力 + 审美洁癖），可能把"我需要的"等同于"知识工作者需要的"
5. **新工具综合症** —— 看到新协议（Skills、MCP、Agent SDK）就想立刻接入，可能把"接入"误判为"价值"

**+ 情绪根源（这一条最重要）**：

> **FOMO（错失焦虑恐惧）驱动的囤积循环**：
> 看到好用的 skill / 方法论 / 视觉设计 → 先收藏/下载下来 → 真到需要用时无法精准找到，甚至忘记自己存过 → 即使能想起来，也要和 AI 铺垫很久的上下文 → 焦虑加剧 → 更激进地囤积。
>
> 这个工具的核心目标是同时打破两层：
> - **L1（人类层）**：让用户日常主动使用，巩固或主动复盘自己的知识元认知
> - **L2（AI 层）**：通过 skill 化，让任意 AI 在 vibe coding 或聊天时"懂我的底层元认知"，更精准抓住我的需求

## Phase 2 · Look Outward (向外看)

### 2.1 谁经历这个问题？（Who experiences it?）

**主体**：AI 时代的**中段**知识工作者
**身份范围**：不限于 AI PM 转型者，扩展到所有"已重度使用 AI、但还没建立自己认知秩序的知识工作者"

**典型场景**（4 个全部成立）：
1. **vibe coding 场景** —— 用 Cursor/Claude Code 做新功能，模糊感觉"之前看过相关方法/组件"，找不到，AI 从零生成质量打折
2. **新需求启动场景** —— 知道"应该用某个方法论"但记不清细节，只能凭印象问 AI，缺方法论尺子兜底，自我怀疑加剧
3. **跨设备/跨工具场景** —— A 工具收藏的内容到 B 工具完全调不出来；浏览器收藏的网页对 Claude Code 完全不可见
4. **复盘/学习场景** —— 想主动回顾"最近这个月学了什么"，打开各种工具看一圈拼不出完整图景，挫败感比不打开还强

**真正的代价**：
> 不止"事情没做好"，更深一层是**身份焦虑**——
> "**作为知识工作者，我所有的积累是不是都在白积累。**"
>
> 这个焦虑在 PRD §2.3 里原本是"AI PM 身份焦虑"，新定位下泛化为"知识工作者身份焦虑"——根基没变，范围更广。

---

### 2.2 谁还有 / 谁没有这个问题？（Who else / who doesn't?）

**有这个痛点的人群**：
- ✅ 重度 AI 工具用户
- ✅ 跨界并行学习者
- ✅ 研究者 / 创作者

**他们的现状**：
> 大部分基本放弃，纯靠个人主观建立文件夹整理。

**没有这个痛点的人**：
- ❌ 不怎么用 AI 的人（不是目标用户）
- ❌ 用 AI 已经炉火纯青、有了自己的一套个人记忆沉淀机制的专家（不是目标用户）

**人群定位**：
```
👶 不用 AI → 没痛点（不是你的用户）
🧗 重度但混乱 → 痛点最深（你的核心用户）⭐
🧘 已成专家有自己一套 → 已自洽（不是你的用户）
```

> 💡 这是一个**漏斗中段**人群——他们已经踏进 AI 时代但还没建立自己的秩序。这个人群在未来 1-2 年会**爆炸式增长**，因为越来越多人正在从"不用 AI"迁移过来，但只有极少数能自己摸索出秩序。**这是一个时间窗口正在打开的市场。**

---

### 2.3 谁被忽略 / 谁从问题存在中获益？（Who's left out / who benefits?）

**被忽略的群体（必须照顾到）**：

1. **未来的我（疲于维护）**
   - 现状：现在的你享受打磨工具，但 6 个月后的你会厌倦维护
   - 解药：**Skill 范式 = 使用即维护**。AI 通过 skill 调用资产时，同时被授权追加新信息，资产在被使用的过程中自动更新和成长

2. **不写代码的同行**
   - 现状：同样有此痛点的非技术知识工作者无法 self-host Vite 项目
   - 解药：**桌面端 + 一份外挂 skill**。封装为 Mac/Windows 桌面应用，用户配置 LLM key 即可；工具暴露一份 skill，让用户挂载到任意 vibe coding 工具（Cursor / Claude Desktop / Cherry Studio）

**结构性受益方（即"问题继续存在"的隐性获利者）**：

| # | 获益方 | 获益方式 |
|---|---|---|
| 1 | SaaS 云端锁定 | 用户资产越散落、越无主权，云端越能锁定你 |
| 2 | 笔记 silo（Notion/Obsidian/Raycast 等）| 它们靠"提供一个 silo"获益，恰恰不希望被打通 |
| 3 | Token 经济 | 用户每次都从零铺垫上下文 = 每次都消耗更多 token，meta 层是赢家 |

> 💡 **三者本质同一**：让用户的资产保持碎片化、保持依赖中心化服务的整个商业范式。这意味着 CCL Atlas 必须有一个**反向价值主张**作为产品 DNA：
>
> > "你的资产 100% 在你本地。它服务你和你授权的 AI，而不是任何中心化平台。它让你越用 AI，越自由——而不是越依赖。"

---

### 2.4 关键差异化判断 · vs 大厂 Cowork 类工具

> 这是 framing session 中最关键的战略洞察之一。

**大厂结构性能做的事**：
- ✅ 本地文件索引感知（OS 级权限 + 规模 + 分发）
- ✅ 被动元信息扫描
- ✅ 跨应用上下文打通

**大厂结构性不会做的事**：
- ❌ 主动方法论沉淀（结构化、有骨架、有筛选）
- ❌ 视觉冲击驱动的复盘体验（它们要 utility，不要 delight）
- ❌ "认知主权"的归属感（vs 监控感）

**为什么大厂不会做？三条结构性原因**：

1. **服务规模化的强迫症** —— 大厂任何功能必须服务 1 亿用户才划算，而"主权层 + 个性化元认知"天然反规模化
2. **数据飞轮的反向利益** —— 让用户数据"留在本地、由用户主权"，等于主动放弃自己最大的护城河
3. **品牌定位的反向锁** —— Cowork 想做的是"AI 帮你工作"，本质是**降低用户的存在感**；CCL Atlas 要做的是"AI 让你更懂自己"，本质是**强化用户的存在感**——两者方向相反，不是先后问题

**战略推论**：

| | 错误的 V1.5 路线 | 正确的 V1.5 路线 |
|---|---|---|
| 心态 | 自己造一个 mini 版的 Cowork | 假设 Cowork 已经存在，我接住它 |
| 投入 | 死磕本地文件扫描的工程难题 | 死磕"被动 → 主动"的转化体验 |
| 风险 | 6 个月后 Cowork 上线，工程白做 | 6 个月后 Cowork 越普及，你越有用 |

> 这意味着 CCL Atlas 的位置不只是"窗口期"，而是**结构性安全的小生境**——大厂越长大，越不可能下场抢，因为 ROI 算不过来。

## Phase 3 · Reframe (重构)

### 3.1 Problem Statement (v2 · 终版)

> **AI 时代的中段知识工作者**——那些已经重度使用 AI、但还没建立自己认知秩序的人——正陷入一个 **FOMO 驱动的囤积循环**：他们不断收藏 skills、方法论、网页、本地素材，但这些资产**对自己不可见、对 AI 不可用**。关键时刻找不到，即使想起也要从零和 AI 铺垫上下文，最终所有积累都退化为**"加剧焦虑的死库存"**。
>
> 真正的代价不是"事情没做好"，而是**一个隐性的身份危机**：你越用 AI，越发现自己的积累不属于自己；你越想"AI 懂我"，越要每次重新解释自己是谁。**长此以往，知识工作者的元认知主权会在不知不觉中外包给中心化平台和 token 经济。**
>
> 这个问题至今没被解决，是因为它要求一个**没有任何现存产品形态能同时承载的双层架构**：
>
> - **L1 · 主动复盘层（让人类用）** —— 视觉冲击驱动的元认知地图，让用户日常主动打开、看见自己拥有什么、产生归属感和心理底气
> - **L2 · 被动接口层（让 AI 用）** —— 一份统一的、可被任意 vibe coding 工具挂载的 skill，让 AI 在对话/编码中自动调用用户的元认知，并在使用过程中协助维护
>
> 两层缺一不可：**只做 L1 = 又一个 Notion**（资产是死的，AI 用不上）；**只做 L2 = 又一个 RAG/MCP 工具**（用户感受不到主权和归属，会停止维护，飞轮转不起来）。两层互锁才能形成"使用即维护"的自驱飞轮。
>
> 而这个**双层架构在结构上是大厂不会做的**：① 它反规模化（每人的认知都不一样）；② 它反数据垄断（资产必须 100% 留在用户本地）；③ 它的产品哲学是"让 AI 强化用户的存在感"，而大厂 Cowork 类工具的方向是"让 AI 替代用户的存在感"——两者南辕北辙，不会撞车。
>
> 因此这是一个**结构性安全的小生境**，并非短暂的窗口期：大厂越长大，越不可能下场抢；而目标人群（AI 中段使用者）正在爆炸式扩张。**问题不是"能不能做"，而是"该谁来做"，以及"现在不做，5 年后所有人都会后悔自己的认知资产被悄悄吃掉"。**

---

### 3.2 How Might We (终版 · A 哲学 + B 形态 + C 动作)

> **How might we** 为 AI 时代的中段知识工作者打造一个**本地优先的桌面工具**，把碎片化的个人资产转化为一份**数据主权 100% 留在自己电脑里**的"主权元认知层"——既能日常自我复盘、又能被任意 AI agent 调用——
>
> **具体到日常**：当我想到"我之前好像有过一个相关的 [skill / 方法论 / 网页 / 文档]"时，**能在 30 秒内通过视觉地图找到它**；当我和 AI 对话或 vibe coding 时，**AI 能自动调用它，并在使用过程中协助我维护它**——
>
> **as we aim to** 终结 FOMO 驱动的囤积循环，让"使用越多 → 资产越活 → AI 越懂我 → 用户越想用"的飞轮自然转起来，**让用户越用 AI 越自由，而不是越用 AI 越依赖中心化平台**。

## Step 2 · JTBD 跑完结果

> **Session 日期**: 2026-04-08（紧接 Step 1 同一天）
> **方法**: Clayton Christensen JTBD + Switch Interview（单用户自我访谈模式）
> **核心问题**: 用户雇佣 CCL Atlas 想完成的，是 1 个 Job 还是 N 个 Job？
> **结论**: **1 个伞下 Job，15 条子 Job**，且必须 **L1+L2 双层服务** 才能闭环

### Umbrella Job（伞下 Job · 用户真正在雇佣工具完成的事）

> *帮我让囤过的每一样东西都能在我需要时真的用上，让我作为知识工作者**心里有数**——不再为 AI 时代的信息焦虑买单。*

### 15 条子 Job 的完整拆解

**功能层（F · Functional Jobs）**

| # | Job | 服务层 |
|---|---|---|
| F1 | 在工作中段卡住时，30 秒内导航到当下场景最相关的方法论/skill | L1+L2 |
| F2 | 在 vibe coding 时，用"审美/直觉"语言反查到专业词汇 + 已收藏视觉示例 | L1+L2 |
| F3 | 让 AI 在对话/编码时自动调用我的元认知，不需要重新铺垫上下文 | 仅 L2 |
| F4 | 让散落在 N 个旧工具/N 个文件夹的旧资产被结构化、被索引、被复活 | 仅 L1 |
| F5 | 每天/每周能主动打开一个地方，看见积累了什么、用过什么、哪些在沉睡 | 仅 L1 |
| **F6 ⭐** | **极低摩擦地把新流入资产沉进系统，免得又变成下一个 silo** | **L2 主导（新认知）** |

**社交层（S · Social Jobs）**

| # | Job | 服务层 |
|---|---|---|
| **S1 ⭐** | **在评审/对话中有方法论的尺子——能主导而不是被牵着走** | L1+L2 |
| S2 | 在和 AI 协作时让 AI 表现出"懂我"（社交资本） | 仅 L2 |
| S3 | 在跨界成长圈子里被认作"有自己一套秩序的人" | L1 外显 |

**情感层（E · Emotional Jobs）**

| # | Job | 寻求/逃离 | 服务层 |
|---|---|---|---|
| E1 | "心里有数"的踏实感 | 寻求 | L1 主导 |
| E2 | "AI 终于懂我"的被看见感 | 寻求 | L2 主导 |
| E3 | 逃离"我学过但用不上 → 我不配"的身份焦虑 | 逃离 | L1+L2 |
| E4 | 逃离 FOMO 囤积悖论（积累越多越焦虑）| 逃离 | L1+L2 |
| E5 | 逃离"每次和 AI 协作都从零开始"的疲惫感 | 逃离 | L2 主导 |
| **E6 ⭐** | **知识工作者的本地主权感（我的认知是我的，不被云端吃掉）** | 寻求 | L1+L2+桌面端 |
| **E7 🆕⭐** | **从 AI 隐性记忆黑箱中夺回可见性和跨 Agent 主权** | 寻求 | L1 可见 + L2 跨 Agent |

### 用户自选的 Top 3 + 1 必杀 Job

> 在 Q2 的强度排序题中，用户只选了 **F6 / S1 / E6**。这个选择模式暴露了 4 个战略级信号：

1. **三层各选 1 条** → 证明 CCL Atlas 不能降维成"功能工具"或"情绪工具"，必须三层都打中
2. **跳过了 F1/F3/E3/E4 这些"看似核心"的 Job** → 这些是后果级症状，用户潜意识选的 F6/S1/E6 是根因级动因
3. **从"逃离"转向"寻求"**（没选 E3/E4，选了 E6）→ CCL Atlas 是**主权工具**，不是反焦虑工具
4. **三条构成自驱飞轮**：E6 寻求主权感 → 激发使用 → F6 极低摩擦沉淀 → S1 有方法论的尺子 → 对话主导 → 每次使用强化主权感 → E6

经 Q3 的追问补齐后，最终锁定为 **Top 3+1**：

| # | Job | 服务层 |
|---|---|---|
| **F6** | 极低摩擦沉淀新资产（**手段改为 agent-driven，不押注人类自律**）| L2 + 后台 |
| **S1** | 有方法论的尺子，在对话中主导 | L1+L2 |
| **E6** | 知识工作者的本地主权感 | 桌面端+L1+L2 |
| **E7 🆕** | 从 AI 隐性记忆黑箱中夺回可见性和跨 Agent 主权 | L1 可见+L2 跨 Agent |

### 必杀痛点

> **"人类长期必然偷懒 + AI 的记忆都是黑箱、碎片、不可控"**
>
> 这两句话加起来构成 V1.5 必须同时解决的**双重杀手**：
>
> - 如果只解决前者（降低人类维护摩擦），但仍然赌用户自律 → 6 个月后用户放弃维护，工具变成墓地
> - 如果只解决后者（做一个更好的 AI 记忆层），但没有可见性和主权 → 等于又做了一个 Claude Memory，用户凭什么信任你

## 🔄 L1/L2 模型 v2（基于 Step 2 的重大认知升级）

> ⚠️ 这一节**推翻并替换** Phase 1 Q3 里的初版 L1/L2 模型。初版认为"L1 是发动机，L2 是放大器"；v2 模型是反的。

### 初版模型（Step 1 Phase 1 Q3 提出，已废弃）

```
L1 = 人类主动复盘层（看见、沉淀、整理）  ← 主角
L2 = AI 被动调用层（skill 接口）      ← 附属
赌注：用户会主动维护 L1
```

### v2 模型（Step 2 Q3 用户澄清后锁定）

```
L2 = Agent 双向工作层（读 skill + 写 skill）  ← 主角
     · 主语：AI
     · 动作：自动沉淀、自动调用、自动维护
     · 承担：90% 的日常工作量

L1 = 主权观察/校准层（监督 + 修正 + 归档 + 复盘）  ← 监督器
     · 主语：人
     · 动作：巡检、看见、校准、删除、打标签
     · 承担：10% 的周度巡检工作量
     · 提供：心理底气、可见性、主权感

飞轮驱动力：L2 承担所有"日常工作量"，L1 提供"信任基础和可见性"
一旦 L1 消失（用户感觉不到主权），用户会因不信任而"拔插头"
一旦 L2 消失（全靠人类自律），飞轮因维护疲劳而死亡
```

**v2 模型的核心哲学转变**：
> **不押注人类自律，押注 agent 工作量。** 人类只负责"监督 + 校准 + 感到主权"，AI 负责"日常搬砖"。

## 🎯 竞争定位 · 4 象限结构性空缺

> 这是 Step 2 用户洞察打开的一个之前 framing 里没命名过的重大定位。

|  | **Agent 集成** | **无 Agent 集成** |
|---|---|---|
| **用户主权 + 透明** | 🟢 **CCL Atlas（独占空缺）** | Obsidian / Notion / Logseq / Bear |
| **平台闭环 + 黑箱** | Claude Memory / ChatGPT Memory / Cowork | 原始 AI 聊天机器人 |

**四个象限的结构性命运**：

- **左下（Obsidian 类）**：哲学上就是"人类手写笔记"，做不到 Agent 集成
- **右下（纯聊天）**：既无主权也无集成，正在被淘汰
- **右上（Claude/ChatGPT/Cowork Memory）**：商业模式决定必须锁定——"透明+主权"会动摇护城河
- **左上（CCL Atlas）**：**没有任何现存工具占据**，而且占据它不需要打败任何对手——因为其他象限的玩家结构上做不了这件事

**战略推论**：这不是"窗口期小生境"，是**结构性空缺**。CCL Atlas 需要赢得的不是竞争，而是"被需要的时刻"——让中段 AI 用户意识到"我需要一个跨 Agent 的、透明的、本地主权的记忆层"。

## 🌟 北极星 Demo 场景（V1.5 成功的最终验收标准）

> 任何 V1.5 功能候选如果不服务这个场景，都不进入 V1.5。

```
场景一 · 沉淀（Claude Code 自动 write）
  用户：我和 Claude Code 一起解决了一个前端 UI 问题，觉得好
  用户说："把这个沉进去"（或 Claude 主动问"要沉进去吗？"）
  Claude 调用 CCL Atlas 的 write skill（本地 skill / MCP 契约）
    → 读取当前对话的关键片段
    → 按 atom schema 结构化（Claude 自己做，用户不动手）
    → 写入 ~/ccl_atlas/data/atoms/...json
    → 更新索引

场景二 · 调用（Cursor 自动 read）
  用户换到 Cursor 做另一个类似项目
  Cursor 自动（或用户喊一声）调用 CCL Atlas 的 read skill
    → 根据当前上下文检索最相关的 atom
    → 作为上下文注入 Cursor 当前对话
    → 用户看都不用看，Cursor 已经"懂我"

场景三 · 监督（用户周日巡检）
  周日晚上，用户打开 CCL Atlas 桌面 GUI
    → 看见过去一周 agent 自动沉淀了 12 张 atom
    → 其中 3 张被 Cursor 和 Claude Code 共调用了 5 次
    → 用户翻一眼、删掉 1 张不准的、把 2 张打 "核心资产" 标签
    → 关掉，去睡觉
  L1 的全部工作量 = 2 分钟巡检
```

**三个场景的相互关系**：
- 场景一、二是 **L2 承担的日常工作量**（每天 N 次，用户零感知）
- 场景三是 **L1 提供的信任基础**（每周 1 次，用户主动打开）
- 飞轮闭环：L2 越被用 → 沉淀越多 → 场景三越值得看 → 用户信任越稳 → 越愿意让 L2 多做

**V1.5 的验收标准 = 这三个场景能否在一个桌面 app 里跑通**。
