## 0 · 为什么需要 V2.0 的重新 framing

V1.5 解决了"**个人记忆资产无法被自己和 AI 共同高效检索调用**"的根问题,ship 了 L1 (桌面 GUI) + L2 (CLI + skills) 的双层主权架构。但 dogfood 过程中暴露出三个新的、在 V1.5 框架内解不掉的张力:

1. **入口张力** · L2 只收文字 agent 生成的 crystallized 经验,用户现实生活里大量的**碎片**(语音备忘、会议 md、灵感闪现、踩坑笔记)没有顺滑的入口
2. **结构张力** · methodology 骨架固定是对的,但 experience 原子散落在"我的经验"里没有**涌现结构**,用户越存越多就越找不到自己存过什么
3. **相遇张力** · V1.5 的 atlas-read 只返回单一类型的 atom,方法论和亲身经验**从不在同一次回答里相遇**——而这恰恰是用户最需要的那个瞬间

V2.0 的使命:**让方法论和碎片在同一次 AI 调用里相遇,同时把入口扩展到全模态,把结构从静态骨架升级为双骨架涌现**。

## 1 · V2.0 北极星短语(价值主张)

> **把你的认知资产,从 AI 里拿回来。**
>
> 你读过的每一本方法论、你和 AI 聊过的每一次顿悟、你私下写下的每一句踩坑——它们不该每次打开新对话就清零。Atomsyn 把它们变成**你本地的、你自己的、能随时被任何 AI 唤醒的**一层记忆。
>
> 你可以主动打开它,看见自己的认知地图正在长成什么样子,在 AI 的知识洪水里**保持自己的清醒**;你也可以什么都不做,下一次和 Claude、Cursor、Codex 对话时,它们看到的不再是一个空白的你,而是一个**一直在复盘、一直在进步**的你。
>
> *Atomsyn — your meta-cognition, rematerialized.*

这一段是 V2.0 所有功能取舍的最高裁决者。任何新功能在进入 PRD 前,必须先回到这里检查:**它服务于"主动复盘"还是"被动唤醒"?如果两者都不是,就不是 V2.0 的事。**

## 2 · Phase 1 · Look Inward (向内看)

### 2.1 V2.0 要解决的新问题症状

| # | 症状 | V1.5 现状 | V2.0 目标 |
|---|---|---|---|
| A | **全模态碎片无家可归** | 只有 agent 生成的 crystallized experience | CLI + GUI 双端收口,文本/md 先行,音频留接口 |
| B | **骨架静态无法生长** | 24 步法固定 + "我的经验"扁平列表 | 双骨架并存:权威骨架(固定)+ 涌现骨架(动态) |
| C | **方法论与经验永不相遇** | atlas-read 返回单类型 atom | 一次调用同时返回方法论 + 挂在其下的个人碎片 |
| D | **模型配置能力不足** | 单一 LLM 配置,不支持 VLM/ASR/embedding | 多模型多厂商全兼容,多模态管线可配置 |
| E | **品牌与名字受限** | ccl-atlas 是项目代号不是产品名 | 改名 Atomsyn,全链路替换 |

### 2.2 为什么这些问题在 V1.5 没解决?

- **时间窗口**:V1.5 的任务是先把主权双层架构立住,把 ship 的门槛抬到"能给自己用"。多模态和双骨架是建立在这个地基之上的第二层。
- **认知递进**:V1.5 ship 之后用户(我)才真实体会到"碎片无处存"的痛感,这个痛感是 dogfood 出来的,不是 framing 出来的。
- **技术依赖**:双骨架需要语义对齐能力,语义对齐需要 embedding 或 LLM 分类能力,LLM 分类能力需要模型配置升级——**这是一条强依赖链**,必须按 M1 → M2 → M3 的顺序做。

### 2.3 V2.0 的新偏见(我要警惕自己的)

1. **全模态贪心** — 很想一次做完文本+会议总结+音频+图像,但 ASR 是独立大坑,V2.0 必须克制只做文本+md
2. **动态骨架崇拜** — 很容易被"全涌现"诱惑,彻底放弃固定骨架。但这会破坏 V1.5 的"方法论是稳定锚"哲学。**双骨架并存**才是对的
3. **schema 过度设计** — 面向跨行业知识工作者时,很容易把 schema 设计成"什么都能装"的大杂烩。要守住**4 个核心维度 (role/situation/activity/insight_type) + 开放 tags** 的最小集
4. **改名拖延** — 改名工作很无聊没有成就感,很容易想"先做功能最后改"。但路径迁移越晚代价越大,**M0 必须是第一件事**
5. **"相遇"的过度承诺** — 很容易把"相遇"做成一个炫技的大弹窗。它应该是**低调的、信任的**:atlas-read 返回里默默带上一段"你两个月前也遇到过类似的",让用户自己"哦"一声

## 3 · Phase 2 · Look Outward (向外看)

### 3.1 用户画像的扩展

V1.5 的目标用户是"AI 时代的中段知识工作者",V2.0 明确扩展到**跨行业的中度-重度 AI 知识工作者**:

- 产品经理 / 咨询顾问 / 设计师 / 研究员 / 律师 / 医生 / 创业者 / 作家 / 教师 / 辅导顾问……
- **共性不是"行业"**,而是"**都在用脑子换产出 + 都重度用 AI + 都有自己的方法论和踩坑库**"
- **反例**:纯操作类工作(数据录入、客服)、非 AI 用户、只用 AI 生成文案不用 AI 思考的轻度用户

这个扩展直接决定了 D5 的 schema 设计(下一节)不能用 `industry` 作为第一维度,因为跨行业的认知工作才是真相。

### 3.2 V2.0 北极星 Demo 场景

> 两个月前的一个下午,用户在和客户开完一个 B 端产品的访谈后,在手机备忘录里随手记了一段:
>
> > "访谈对象嘴上说'我们最缺'一个 AI 助手,但聊到一半他自己承认:'其实我们现在最痛的是交接文档没人维护。'——原来客户嘴里的'最缺'不是真'最痛'。下次访谈要刻意在 15 分钟后再问一次同样的问题,看答案会不会变。"
>
> 他当天晚上用一条命令:
>
> ```
> $ atomsyn-cli ingest --text "访谈对象嘴上说..."
> ✓ Classified: role=研究 · situation=访谈 · activity=倾听
> ✓ Insight type: 反直觉
> ✓ Linked to methodology: atom_jtbd_five-whys (conf 0.82)
> ✓ Saved to experience/fragment: atom_exp_interview-stated-vs-revealed-need_1775999999
> ```
>
> 两个月后,他在 Cursor 里用 Atomsyn 的 vibe coding 场景开启一个新项目,问:"帮我起草一份产品用户访谈提纲。"
>
> Cursor 调了 `atomsyn-read --query "用户访谈提纲"`。它的回答里除了 JTBD 五问方法论的要点,还多了一小段:
>
> > 📎 **你两个月前记下过**:"访谈对象嘴上说的'最缺'不一定是真'最痛',下次要在 15 分钟后再问一次。" — 标记为**反直觉**洞察,建议纳入本次提纲的验证环节。
>
> 用户在屏幕前愣了两秒,然后把这条"15 分钟后再问"写进了访谈大纲。
>
> **这两秒就是 V2.0 的全部价值。**
>
> 这是相遇。不是弹窗,不是炫技,是一个安静的、你自己都快忘了的瞬间,在你需要它的时刻**被你自己的 AI 递回给你**。

### 3.3 竞品与结构性护城河

- **Notion AI / Obsidian Copilot** — L1 强 L2 弱,资产躺着不会被 AI 主动消费
- **ChatGPT / Claude memory** — L2 强但完全非主权,用户数据在云端
- **Mem / Rewind** — 全量记录但没有"骨架 + 方法论"的结构化层
- **Cursor Rules / Claude skills 官方市场** — 只有方法论 skill,没有个人经验回路

V1.5 已经验证的结构性小生境(反规模化/反数据飞轮/哲学相反)在 V2.0 继续成立,且**双向蒸馏的相遇闭环**是一个更难被大厂抄袭的点——因为它要求厂商放弃云端记忆模型,而这和他们的商业模式直接冲突。

## 4 · Phase 3 · Reframe (重构)

### 4.1 V2.0 的核心洞察(接续 V1.5 的 12 条,新增第 13-17 条)

13. **碎片是方法论的反面,但它们互为镜像。** 方法论是"别人验证过的共性",碎片是"你自己验证过的个性"。只有两者相遇才能真正帮助决策。
14. **入口决定一切。** 只要写入门槛比"打开 Notion 新建一页"高,用户就会选择不写。`atomsyn-cli ingest` 必须是**单行命令 + stdin 友好 + 零思考延迟**。
15. **静态骨架是锚,动态骨架是河。** 权威方法论的结构稳定性不能让位,但个人经验的涌现性也不能被扼杀。双骨架是两者的平衡解。
16. **语义对齐不是技术问题,是哲学防线。** 没有对齐,用户两个月内会造出"产品创新" vs "产品方法论" vs "产品思考" 三个语义重复的骨架,然后彻底放弃这个工具。对齐必须发生在**写入时**,由 skill/GUI 两端各自引导,不靠后期清理。
17. **相遇不能炫技。** 它的美学是"低调的、可被忽略的、但一旦被注意到就无法忽视的"。atlas-read 的返回格式必须让用户能一眼跳过,也能一眼抓住。

### 4.2 4 象限定位(V2.0 更新)

```
                  高主动性(人类主动复盘)
                         ↑
                         │
         Atomsyn V2.0 ───┼───→
         (GUI 记忆花园)    │
                         │
高结构化 ←───────────────┼───────────────→ 低结构化
(双骨架)                  │               (碎片池)
                         │
                         │
         Atomsyn V2.0 ───┼───→
         (L2 唤醒相遇)     │
                         │
                         ↓
                  高自动化(AI 自动调用)
```

V2.0 同时占据 4 个象限。这在 V1.5 是 L1/L2 双层,V2.0 进化成 **结构化/涌现 × 主动/自动** 的四象限矩阵。这个矩阵决定了下一节 PRD 的 milestone 划分。

### 4.3 核心四维 schema(D5 决策的固化)

所有通过 `atomsyn-cli ingest` 或 GUI 上传的碎片,必须生成以下 meta:

```jsonc
{
  // 核心四维(完全开放 · 种子候选表仅作 autocomplete 提示,不强制命中)
  "role": "研究 | 产品 | 设计 | 咨询 | 决策 | 创作 | 协作 | 学习 | 教学 | 辅导 | 自我管理 | ...",
  "situation": "会议 | 访谈 | 独立思考 | 阅读 | 对话AI | 复盘 | 踩坑当下 | 灵感闪现 | 冲突 | 决策关口 | ...",
  "activity": "分析 | 判断 | 说服 | 倾听 | 试错 | 验证 | 综合 | 表达 | 拒绝 | 妥协 | 观察 | ...",

  // 半封闭枚举(唯一) — 决定唤醒层的呈现方式
  "insight_type": "反直觉 | 方法验证 | 方法证伪 | 情绪复盘 | 关系观察 | 时机判断 | 原则提炼 | 纯好奇",

  // 完全开放
  "tags": ["关键词1", "关键词2"],

  // 语义对齐产物(M4 自动填充)
  "linked_methodologies": ["atom_id_..."],

  // 可选上下文
  "context": {
    "domain_hint": "自由文本,例如 'B端竞品分析' / '家庭沟通'",
    "project_ref": "atom 或 project id"
  }
}
```

**关键设计约束**:

- **没有 `industry` 字段** — 跨行业知识工作者的共性不是行业
- **四维完全开放,骨架完全开放** — 种子表只是 autocomplete 提示,用户和 agent 都可以自由新增任意分类词和任意新骨架。不搞"半封闭"避免用户感到被束缚
- **`insight_type` 是唯一例外的半封闭枚举** — 因为它决定 UI 和检索呈现(**情绪复盘**会进私密视图,不被 skill 公开检索,只在 GUI 显式查询时露出),新增这一维的新值需要同步更新 `src/lib/insightColors.ts` 色码表,成本可控
- **种子候选表位置**:`data/taxonomy/seed.json`,可在 GUI settings 中编辑
- **语义对齐的责任不在 schema,而在写入引导**:CLI/skill 端在 prompt 里要求 agent 先 find 再写;GUI 端用 autocomplete + 弹确认。两端各自守好自己的入口,schema 保持开放

## 5 · V2.0 继承自 V1.5 的不可动摇哲学(硬约束)

这些是 V1.5 framing session 沉淀的 DNA,V2.0 的任何设计动作都**必须先回到这里检查**:

1. **本地 100%,主权第一** — 用户数据从不离开本地。LLM/VLM/ASR/Embedding 的 key 永远只在 localStorage,永不进 git。
2. **L1 + L2 双层缺一不可** — V2.0 的双骨架 + 多模态入口 + 相遇层必须在两层上同时成立。
3. **大厂结构性不会做** — V2.0 的"双向蒸馏相遇闭环"强化了这个结构性护城河。
4. **使用即维护** — 碎片被 ingest、被相遇、被用户在 GUI 里手动合并的那一刻,就是它被维护的那一刻。
5. **CLI-first** — 所有写入(包括多模态解析产物)必须经过 `atomsyn-cli ingest`,agent 永远不直接写磁盘。
6. **设计语言硬约束** — Linear/Raycast 变体 + Inter + 玻璃态 + spring + 4 层渐进披露 + 浅色优先 + 全暗色对照。任何 UI 任务必须先读 mockups + 派 code-reviewer 审查。

## 6 · V2.0 不做的事(范围护栏)

为了守住价值主张和避免 V2.0 蠕变为大杂烩,**明确列出 V2.0 不做**:

- ❌ **语音 ASR 管线** — 只在模型配置里留接口,不实现 ingest 通道(V2.x)
- ❌ **图像/VLM 解析** — 同上,配置预留,V2.x 再做
- ❌ **生活/情感/心理疗愈领域扩展** — 会稀释价值主张(V3+)
- ❌ **云同步** — 主权哲学的硬约束
- ❌ **账号系统** — 主权哲学的硬约束
- ❌ **Embedding 语义检索替代 keyword 检索** — V2.x,V2.0 的语义对齐用 LLM prompt 做就够
- ❌ **Merge atom 的真实合并逻辑(非 stub)** — V1.6 backlog 延续
- ❌ **GitHub Releases 抓取** — V1.6 backlog 延续

## 7 · V2.0 对下游 PRD 的约束

本 framing 文档对 `docs/PRD-v2.0.md` 的强约束:

1. **PRD 的 milestone 划分必须是 M0 → M1 → M2 → M3 → M4 → M5**,顺序不可调(因为是强依赖链:改名 → 模型配置 → 解析管线 → 双骨架 GUI → 相遇层 → 稳定性固化)
2. **每个 milestone 必须有"S4 稳定性验收门"**,不单独成为 milestone
3. **所有新代码中的 CLI 名必须是 `atomsyn-cli`**,不得出现 `atlas-cli` 新引用(旧代码在 M0 统一替换)
4. **D5 schema 固化为 `skills/schemas/experience-fragment.schema.json`**,四维 + tags + insight_type + 可选 context,**字段集保持稳定,但字段值完全开放**
5. **D4 首次启动迁移弹窗的拷贝文案必须体现仪式感**,让用户"看见它发生"
6. **V2.0 北极星 Demo 场景必须可被 M4 完整复现**,作为 M4 的唯一验收标准
7. **V2.0 整体 Exit Criteria 是业务闭环而非工程指标**:"在 Claude Code + Cursor 里通过 `atomsyn-cli` 能完整走通沉淀 + 唤醒双向闭环",这条凌驾于任何 milestone 的技术验收门
8. **GUI nav 中"我的经验"必须在 M3 彻底移除并降解进记忆花园**,不做长期兼容路由(V2.0 MVP 期用户只有本人一位,允许 breaking)
9. **M3 的 Fragment Card 视觉必须严格对照 `docs/mockups/atom-card.html` + `CLAUDE.md` 视觉语言**,D5 schema 的 4 个字段必须在 4 层渐进披露中有明确的分层呈现规则
